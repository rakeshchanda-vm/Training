# Internal: Workspace sandbox providers (Bubblewrap + gVisor)

This document describes the non-KVM sandbox backends for `WorkspaceSession`:

- `bubblewrap_process` — fast per-command Linux process sandbox
- `gvisor_container` — stronger container sandbox via Docker + `runsc`

Use these on hosts without `/dev/kvm`. They are **not** microVM-grade multi-tenant isolation.

## Provider comparison

| Backend | Isolation | KVM required | Typical use |
|---------|-----------|--------------|-------------|
| `local_fs` | None | No | Dev only |
| `ephemeral_fs` | Disposable workspace | No | Dev / CI |
| `bubblewrap_process` | Per-command namespaces | No | Fast local/dev shell sandbox |
| `gvisor_container` | gVisor kernel sandbox | No | Stronger dev/prod-ish container boundary |
| `microvm` | Kata/Firecracker VM | Yes | Production VM boundary |
| `postgres_vfs` | DB-backed files only | No | File tools without shell |

## API (typed settings)

```python
from andromeda.workspace import (
    BubblewrapProcessSettings,
    FileSeed,
    GVisorContainerSettings,
    WorkspacePolicy,
    WorkspaceSession,
)

# Bubblewrap — per-command bwrap sandbox
session = WorkspaceSession.create(
    backend="bubblewrap_process",
    root="./workspace",
    settings=BubblewrapProcessSettings(),
    policy=WorkspacePolicy(enable_shell=True),
)

# gVisor — long-lived Docker container with runsc
session = WorkspaceSession.create(
    backend="gvisor_container",
    settings=GVisorContainerSettings(image="python:3.12-slim"),
    policy=WorkspacePolicy(enable_shell=True),
)
```

Agents use `session.tools()` — same as other backends (`read_file`, `shell`, etc.).

## Bubblewrap setup

Ubuntu / Debian:

```bash
sudo apt-get update
sudo apt-get install -y bubblewrap
bwrap --version
cat /proc/sys/user/max_user_namespaces
```

Amazon Linux:

```bash
sudo yum install -y bubblewrap
bwrap --version
```

Integration tests (opt-in):

```bash
export ANDROMEDA_BUBBLEWRAP_TEST=1
pytest tests/test_workspace_session.py::test_bubblewrap_integration_when_configured -q
```

### Bubblewrap behavior

- Each `shell` invocation runs a fresh `bwrap` command generated by the app
- Workspace mounted writable at `/workspace`
- Network disabled by default (`BubblewrapProcessSettings.network=False`)
- Host env is **not** inherited (`build_sandbox_env`)
- Background shell tools are disabled in v1

## gVisor setup

Install gVisor (`runsc`) using the [official gVisor installation docs](https://gvisor.dev/docs/user_guide/install/) for your OS.

Configure Docker to use the `runsc` runtime, then verify:

```bash
docker run --rm --runtime=runsc --network=none alpine echo ok
```

Integration tests (opt-in):

```bash
export ANDROMEDA_GVISOR_TEST=1
export ANDROMEDA_GVISOR_IMAGE=python:3.12-slim
pytest tests/test_workspace_session.py::test_gvisor_integration_when_configured -q
```

### gVisor behavior

- One long-lived container per session (`sleep infinity`)
- Workspace host directory bind-mounted to `/workspace`
- `--network none` by default
- No Docker socket, SSH, or cloud credential mounts
- Container removed on `session.cleanup()`

Production images should use immutable digests:

```text
python:3.12-slim@sha256:<digest>
```

## Smoke test

```python
from andromeda.workspace import (
    BubblewrapProcessSettings,
    FileSeed,
    WorkspacePolicy,
    WorkspaceSession,
)

session = WorkspaceSession.create(
    backend="bubblewrap_process",
    root="./tmp/ws-bwrap",
    settings=BubblewrapProcessSettings(),
    seed=[FileSeed(path="hello.py", content="print('hello from bwrap')\n")],
    policy=WorkspacePolicy(enable_shell=True),
)
tools = session.tools()
result = tools["shell"].invoke({"command": "python3", "argv": ["hello.py"]})
print(result)
session.cleanup()
```

## When `/dev/kvm` is unavailable

If Kata/microVM setup is blocked (common on standard EC2), use:

1. `bubblewrap_process` for fast command sandboxing during development
2. `gvisor_container` when you need a stronger container boundary without KVM

See also: [nerdctl-kata-setup.md](nerdctl-kata-setup.md)

## Security notes

- Deny lists and path checks are defense-in-depth; the real boundary is sandbox configuration
- Do not mount Docker socket, credentials, or host project roots
- Network is off by default; enable only when required
- These providers are safe-by-default but not equivalent to microVM isolation
